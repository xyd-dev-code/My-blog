## 项目背景

**电脑商城**是我下学期的课程项目，从 2025-06-10 项目初始化到 2025-07-29 完结，历时约两个月。它不是一个 demo，而是一个**完整可上线的电商系统**：用户注册登录、商品浏览、购物车、下单、支付、后台管理，每个环节都跑通了。

为什么单独写一篇综述？因为 5 篇 Day X 日志分散在 posts 里，单独看每一篇只能看到"当天做了什么"，看不到项目全貌。这篇是站在终点的回顾——架构、决策、教训。

## 一、技术栈

| 层 | 技术 | 选型理由 |
|----|------|----------|
| 后端框架 | Spring Boot 2.7 | 学校指定，生态成熟 |
| ORM | MyBatis-Plus | 比 MyBatis 省一半 SQL 代码 |
| 数据库 | MySQL 8.0 | 主存储 |
| 缓存 | Redis | 购物车 / 验证码 / Session |
| 鉴权 | JWT + 自写拦截器 | 无状态，适配前后端分离 |
| 文档 | Swagger (Knife4j) | 接口联调 |
| 部署 | Docker Compose | 一次构建，多机运行 |
| 前端 | Vue 2 + Element UI | 课程要求 |

## 二、模块拆分

```
mall/
├── mall-common/      # 通用工具、常量、异常
├── mall-pojo/        # Entity、DTO、VO
├── mall-mapper/      # MyBatis-Plus Mapper
├── mall-service/     # 业务逻辑
├── mall-controller/  # HTTP 接口
├── mall-admin/       # 后台管理
└── mall-frontend/    # Vue 2 页面
```

经典 7 模块 Maven 工程，`mall-common` 依赖最底层，`mall-controller` 依赖最上层，避免循环依赖。

## 三、数据模型

7 张核心表：

| 表 | 说明 |
|------|------|
| `tb_user` | 用户（含密码 BCrypt 加密）|
| `tb_product` | 商品 |
| `tb_category` | 商品分类（树形）|
| `tb_cart` | 购物车 |
| `tb_order` / `tb_order_item` | 订单主表 + 明细 |
| `tb_address` | 收货地址 |

ER 关系：`user 1—N order`、`order 1—N order_item`、`user 1—N cart`、`product 1—N cart / order_item`。

## 四、关键决策

### 4.1 为什么用 Redis 存购物车

购物车有 3 个特性：

1. **频繁读写**（每个商品详情页可能都改）
2. **实时性弱**（用户关电脑回来还能看到）
3. **数据结构简单**（key=userId, value=Hash）

MySQL 每次写都要落盘，扛不住并发；直接放内存又没法跨进程。所以选 Redis。

具体实现：

```java
// 购物车加入商品
redisTemplate.opsForHash().put(
    "cart:" + userId,
    String.valueOf(productId),
    String.valueOf(quantity)
);
// 设置 30 天过期
redisTemplate.expire("cart:" + userId, 30, TimeUnit.DAYS);
```

### 4.2 JWT 单 token vs 双 token

跟 zhikao-cloud 不同，**商城项目用单 token，2 小时过期**。理由：

- 用户场景：逛商城 → 下单 → 走人，单次会话不超过 2 小时
- 双 token 复杂度高、课程项目没必要

如果用户登录后过期，提示"会话过期，请重新登录"。

### 4.3 订单创建的事务边界

订单创建涉及**多张表写**：

```java
@Transactional
public Order createOrder(Long userId, List<CartItem> items) {
    // 1. 扣库存
    productMapper.decreaseStock(productId, quantity);
    // 2. 写订单主表
    Long orderId = orderMapper.insert(order);
    // 3. 写订单明细
    orderItemMapper.batchInsert(orderItems);
    // 4. 清购物车
    cartMapper.deleteByUserId(userId);
    return order;
}
```

**坑**：第 2、3 步中间抛异常，会导致"库存扣了但订单没生成"。MySQL 默认 `REPEATABLE READ` 隔离级别 + `@Transactional` 默认 rollback-for=RuntimeException 已经能处理大部分场景，但**自调用（A 方法调 B 方法，B 上有 @Transactional）会失效**——必须走 Spring 代理。

### 4.4 后台管理的权限隔离

前后端共用一份 JWT，但角色不同：

- `role=user`：前台用户
- `role=admin`：后台管理

后端拦截器解 token 拿到 role，`/api/admin/**` 路径强制 `role=admin`，否则 403。前端根据 role 路由分流，进不了后台。

## 五、踩坑集锦

### 坑 1：MyBatis-Plus 分页插件失效

**现象**：第 1 页返回全部数据，Page 对象 total 正常但 records 没分。

**根因**：MyBatis-Plus 分页插件需要 `@EnableTransactionManagement` + `PaginationInnerInterceptor` 注册，少一个就失效。

**修复**：
```java
@Configuration
@MapperScan("com.mall.mapper")
public class MyBatisConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}
```

### 坑 2：JWT 密钥泄露

**现象**：开发时把密钥写在 `application.yml` 里，推到 GitHub 后第二天有人来尝试伪造 token。

**修复**：密钥放环境变量，配置文件用 `${JWT_SECRET}` 占位。

### 坑 3：Vue 跨域

**现象**：前端 `localhost:8080` 调后端 `localhost:9090`，CORS 报错。

**修复**：后端写 `CorsConfig`：
```java
.addMapping("/**")
.allowedOriginPatterns("*")
.allowedMethods("GET","POST","PUT","DELETE")
.allowCredentials(true)
.maxAge(3600);
```

### 坑 4：Docker 镜像体积

**现象**：`mall-backend` 镜像 1.2GB，启动慢。

**修复**：多阶段构建 + alpine jre：
```dockerfile
FROM maven:3.8-openjdk-11 AS builder
WORKDIR /app
COPY . .
RUN mvn clean package -DskipTests

FROM openjdk:11-jre-alpine
COPY --from=builder /app/mall-controller/target/*.jar app.jar
EXPOSE 9090
ENTRYPOINT ["java","-jar","/app.jar"]
```

最终 280MB。
