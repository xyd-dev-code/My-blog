## 第四步：日志定位

应用日志在 `/var/log/app/biz-support-backend.log`。

```bash
cd /var/log/app
ls -lh
```

```
-rw-r--r-- 1 app app 500M Dec 28 10:00 biz-support-backend.log
```

一天一个文件，500MB，**日志量很大**。

应用配置里日志输出格式：

```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [requestId=%X{requestId}] %-5level %logger{36} - %msg%n</pattern>
```

**关键** —— **每个请求有 requestId**。工单里应该带了 requestId。

查工单，工单系统有个"日志快查"按钮，一键跳到对应的日志：

```
2025-12-28 09:23:41.234 [http-nio-8080-exec-15] [requestId=abc12345] ERROR
c.x.b.u.controller.UserController - 用户列表接口异常

java.lang.NullPointerException: Cannot invoke "String.length()" because "user.getUsername()" is null
    at com.xxxx.biz.user.service.impl.UserServiceImpl.lambda$listUsers$0(UserServiceImpl.java:78)
    at java.util.stream.ReferencePipeline$3$1.accept(ReferencePipeline.java:197)
    at java.util.ArrayList$ArrayListSpliterator.forEachRemaining(ArrayList.java:1708)
    ...
```

**找到了！**

**异常位置**：`UserServiceImpl.java:78` 行

**错误信息**：`user.getUsername().length()` —— **`username` 为 null，调用 `.length()` 时 NPE**。

**W 哥教我几个查日志的命令**：

```bash
# 看最近 100 行
tail -100 biz-support-backend.log

# 实时跟踪（看现场用）
tail -f biz-support-backend.log

# 找 ERROR 级别
grep "ERROR" biz-support-backend.log | tail -20

# 按 requestId 整链路查
grep "requestId=abc12345" biz-support-backend.log

# 找特定时间段
sed -n '/2025-12-28 09:23:/,/2025-12-28 09:24:/p' biz-support-backend.log
```

我用了**最直接的方法** —— **工单里点"日志快查"**（公司平台功能），**直接定位**。

## 第五步：定位代码

回到工位，本地打开代码仓库，**切到 user 分支**：

```bash
git checkout develop
git pull origin develop
git checkout -b fix/nullpointer-user-list
```

打开 `UserServiceImpl.java`，看 78 行：

```java
@Override
public List<UserVO> listUsers(UserQueryDTO query) {
    // 1. 查数据库
    List<User> users = userMapper.listByCondition(query);

    // 2. 转 VO
    List<UserVO> result = users.stream()
        .map(user -> {
            UserVO vo = new UserVO();
            vo.setId(user.getId());
            vo.setUsername(user.getUsername().length() > 0 ? user.getUsername() : "匿名");  // ← 78 行
            vo.setEmail(user.getEmail());
            // ...
            return vo;
        })
        .collect(Collectors.toList());

    return result;
}
```

**问题**：78 行直接调用 `user.getUsername().length()`，**没判空**。

**为啥 username 会 null**？

W 哥解释了：

> "**业务支撑系统是迁移过来的老系统**。**部分用户是从旧系统同步过来的**，**username 可能没值**。**新系统注册的用户 username 都有**。**所以才'偶发'** —— **老用户触发**。"

**典型的数据兼容问题** —— 新老系统迁移最容易踩。

## 第六个坑：直接 `if (xxx != null)` 是不够的

我第一版修复：

```java
vo.setUsername(user.getUsername() != null ? user.getUsername() : "匿名");
```

**但 W 哥 review 时说**：

> "**还有别的字段也是 string**。**email、phone 都可能 null**。**每个字段判空太丑**。"

正解是写个 `BeanUtils` 工具类，或者用 `Optional` 包装：

```java
vo.setUsername(Optional.ofNullable(user.getUsername()).orElse("匿名"));
vo.setEmail(Optional.ofNullable(user.getEmail()).orElse(""));
vo.setPhone(Optional.ofNullable(user.getPhone()).orElse(""));
```

但更彻底的方案 —— **数据库迁移时给老数据补默认值**：

```sql
UPDATE user SET username = CONCAT('user_', id) WHERE username IS NULL;
UPDATE user SET email = CONCAT('user_', id, '@local') WHERE email IS NULL;
```

**W 哥说这次先在代码层防御，**数据修复**交给 DBA**。

## 第七个坑：加 unit test 验证

W 哥要求："**改完代码要补单测**"。

`UserServiceImplTest.java`：

```java
@SpringBootTest
class UserServiceImplTest {

    @Autowired
    private UserService userService;

    @MockBean
    private UserMapper userMapper;

    @Test
    void listUsers_shouldHandleNullUsername() {
        // 准备：mock 一个 username 为 null 的用户
        User user = new User();
        user.setId(1003L);
        user.setUsername(null);  // 关键：username 是 null
        user.setEmail("test@local");
        when(userMapper.listByCondition(any())).thenReturn(List.of(user));

        // 执行
        List<UserVO> result = userService.listUsers(new UserQueryDTO());

        // 断言
        assertEquals(1, result.size());
        assertEquals("匿名", result.get(0).getUsername());  // 不抛异常，返回"匿名"
    }

    @Test
    void listUsers_shouldHandleNormalUser() {
        User user = new User();
        user.setId(1L);
        user.setUsername("alice");
        user.setEmail("alice@x.com");
        when(userMapper.listByCondition(any())).thenReturn(List.of(user));

        List<UserVO> result = userService.listUsers(new UserQueryDTO());

        assertEquals(1, result.size());
        assertEquals("alice", result.get(0).getUsername());
    }
}
```

跑测试：

```bash
mvn test -Dtest=UserServiceImplTest
```

```
Tests run: 2, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```

**两个测试都过**。

**W 哥强调**：

> "**单测不是为了'过覆盖率'**。**是验证你的修复确实生效**。**以后别人改代码，**单测能保证不破坏现有逻辑**。"

## 第十个坑：写事故复盘文档

```
## 现象
用户列表接口 P99 5 秒，1% 请求返回 500。
每周发生 3 次左右。

## 根因
用户表是迁移老系统的历史数据，部分用户 username 为 null。
UserServiceImpl.listUsers() 第 78 行代码未做 null 防御。

## 修复
- 代码：用 Optional 处理 null
- 测试：补充单元测试覆盖 null 场景
- 数据：DBA 后续补数据（进行中）

## 验证
- 单元测试通过
- 金丝雀 30 分钟监控：P99 120ms，错误率 0.05%

## 反思
1. 数据迁移场景必须考虑字段兼容性
2. 老系统字段值不可信，业务代码必须有防御
3. 单元测试要覆盖 null / 边界场景
4. 建议加 SonarQube 的 S2259 规则（null 检测）
```

W 哥看完说：

> "**复盘写得很清楚**。**根因 + 修复 + 反思都有**。**以后遇到类似问题可以参考**。"

## 第一次修生产 bug 的几个体会

### 1. 生产 bug 排查有"流程"

W 哥总结的标准流程：

```
1. 看监控（Grafana / Prometheus）
2. 查日志（error 级别 + requestId 定位）
3. 定位代码
4. 写修复 + 单元测试
5. 走 Git Flow + Code Review
6. 金丝雀发布 + 监控观察
7. 写事故复盘
```

**每一步都不能跳**。

### 2. 日志 requestId 是真有用

**没有 requestId 我没法定位这次 bug** —— 一天 500MB 日志，**靠时间戳根本找不准**。

**这是公司基础设施的好处** —— **每个细节都有意义**。

### 3. 单元测试不是"应付检查"

**单测确实在保护代码** —— 我写的这次，**改完代码我也用单测验证了才敢上线**。

### 4. 不要照抄线上代码

我电脑商城早期版本也有 `user.getUsername().length()` 的写法 —— **没判空**。

W 哥说：

> "**公司代码一开始也是这样写的**，**后来出 bug 才加防御**。**你以后写代码** —— **所有字段都假设它可能 null**。"

### 5. Git Flow 是"团队协作"

我一个人开发时 Git Flow 看着多余，**但团队协作时是必要的**：

- **不阻塞别人开发**（feature 分支独立）
- **代码有 reviewer**（防止低级错误）
- **上线有节奏**（release 分支冻结）

**这次实习学到最实用的技能 —— 不是 Java，是 Git Flow**。
