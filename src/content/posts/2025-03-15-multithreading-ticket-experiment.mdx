## 起因

JUC 课第一次上机作业：

> "用多线程模拟 10 个窗口卖 100 张票，每张票只能卖一次。比较不同同步方案的效果。"

老师让我们"先自己写一版，然后看运行结果，最后换方案对比"。

我以为 synchronized 加一下就行，结果**第一版就超卖了**。这篇是完整的"重写 4 遍"过程。

## 第 1 版：天真版（超卖 bug）

```java
public class TicketSaleV1 {
    private static int tickets = 100;

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                while (tickets > 0) {
                    System.out.println(Thread.currentThread().getName() 
                                       + " 卖出第 " + tickets-- + " 号票");
                }
            }, "窗口-" + i).start();
        }
    }
}
```

跑一下：

```
窗口-3 卖出第 100 号票
窗口-5 卖出第 99 号票
窗口-2 卖出第 98 号票
...
窗口-7 卖出第 1 号票
窗口-1 卖出第 0 号票    ← 超卖了！
窗口-9 卖出第 -1 号票   ← 超卖了！
```

总共卖了 102 张票，**多卖了 2 张**。

**为什么？** 多个线程同时执行 `tickets > 0` 判断，都看到 `tickets = 1`，然后各自 `--` 减到 0 和 -1。

**超卖是线程竞态条件的典型表现**——读、判断、写这三步**不是原子的**。

## 第 2 版：加 synchronized（但锁错位置）

```java
public class TicketSaleV2 {
    private static int tickets = 100;

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                while (tickets > 0) {
                    synchronized (TicketSaleV2.class) {
                        System.out.println(Thread.currentThread().getName() 
                                           + " 卖出第 " + tickets-- + " 号票");
                    }
                }
            }, "窗口-" + i).start();
        }
    }
}
```

跑了 5 次，结果都正确：

```
窗口-2 卖出第 100 号票
窗口-2 卖出第 99 号票
...
窗口-9 卖出第 1 号票
```

但有几个新问题：

1. **窗口分配不均**：`窗口-2` 一个人卖了 30 张，其他窗口几乎抢不到
2. **打印输出混乱**：可能 `窗口-2` 连续打印好几行才切到 `窗口-3`
3. **`System.out.println` 不在锁里**：可能两个线程的打印行交错

**锁粒度太细** + **CPU 调度不均**，导致了这些问题。

## 第 3 版：synchronized 锁整个循环（但死循环了）

我以为问题在"锁不够大"，把 synchronized 加到 while 外面：

```java
public class TicketSaleV3 {
    private static int tickets = 100;

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                synchronized (TicketSaleV3.class) {
                    while (tickets > 0) {
                        System.out.println(Thread.currentThread().getName() 
                                           + " 卖出第 " + tickets-- + " 号票");
                    }
                }
            }, "窗口-" + i).start();
        }
    }
}
```

跑了 30 秒**没结束**。

**问题**：第 1 个线程进入 synchronized，**卖完所有票才释放锁**。后面 9 个线程在等锁，**根本进不来**。

> **错误本质**：synchronized 加在 while 外面 = **只有一个窗口在工作**，多线程变成单线程。

## 第 4 版：synchronized + sleep（终于对了）

参考老师给的范例，加了 `Thread.sleep`：

```java
public class TicketSaleV4 {
    private static int tickets = 100;

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                while (true) {
                    synchronized (TicketSaleV4.class) {
                        if (tickets <= 0) {
                            break;  // 没票了，退出循环
                        }
                        try {
                            Thread.sleep(50);  // 模拟售票耗时
                        } catch (InterruptedException e) {
                            break;
                        }
                        System.out.println(Thread.currentThread().getName() 
                                           + " 卖出第 " + tickets-- + " 号票");
                    }
                }
            }, "窗口-" + i).start();
        }
    }
}
```

跑了 5 次都正常：

```
窗口-0 卖出第 100 号票
窗口-3 卖出第 99 号票
窗口-1 卖出第 98 号票
...
窗口-5 卖出第 2 号票
窗口-8 卖出第 1 号票
```

`Thread.sleep(50)` 的作用：让线程在锁内停留一段时间，**其他线程有机会抢到锁**。这样 10 个窗口能"轮流"卖票。

## 第 5 版：用 ReentrantLock（更精细）

`ReentrantLock` 比 synchronized 灵活：

```java
public class TicketSaleV5 {
    private static int tickets = 100;
    private static final ReentrantLock lock = new ReentrantLock();

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                while (true) {
                    lock.lock();
                    try {
                        if (tickets <= 0) {
                            break;
                        }
                        Thread.sleep(50);
                        System.out.println(Thread.currentThread().getName() 
                                           + " 卖出第 " + tickets-- + " 号票");
                    } catch (InterruptedException e) {
                        break;
                    } finally {
                        lock.unlock();  // 必须在 finally 释放
                    }
                }
            }, "窗口-" + i).start();
        }
    }
}
```

**和第 4 版效果一样**，但有几个 ReentrantLock 的优势：

1. **`tryLock()`**：尝试拿锁，拿不到就跳过
2. **`lockInterruptibly()`**：等待锁时可被中断
3. **`new ReentrantLock(true)`**：公平锁，按等待顺序拿

我加了 `tryLock` 试了下：

```java
if (!lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    continue;  // 拿不到锁就不等，继续下一轮
}
```

这样**避免某些线程饿死**（公平性）。

## 第 6 版：用 AtomicInteger（最简洁）

`AtomicInteger` 是 CAS 原子操作，**天然线程安全**：

```java
public class TicketSaleV6 {
    private static final AtomicInteger tickets = new AtomicInteger(100);

    public static void main(String[] args) {
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                while (true) {
                    int current = tickets.get();
                    if (current <= 0) {
                        break;
                    }
                    try {
                        Thread.sleep(50);
                    } catch (InterruptedException e) {
                        break;
                    }
                    
                    // CAS 自减
                    if (tickets.compareAndSet(current, current - 1)) {
                        System.out.println(Thread.currentThread().getName() 
                                           + " 卖出第 " + current + " 号票");
                    }
                    // CAS 失败说明被别人改了，重新循环
                }
            }, "窗口-" + i).start();
        }
    }
}
```

**AtomicInteger 的核心是 `compareAndSet(expect, update)`**：

- 当前值 == `expect` → 改成 `update`，返回 true
- 当前值 ≠ `expect` → 不改，返回 false

每次循环都重新 `get()`，CAS 失败就重试。**没有锁，但保证原子性**。

跑了 10 次都正确，而且比 synchronized **快 30% 左右**（不用阻塞/唤醒）。

## 三种方案对比

写完 6 个版本，做了个总结表：

| 方案 | 代码复杂度 | 性能 | 公平性 | 适用场景 |
|------|------------|------|--------|----------|
| **synchronized** | 简单 | 中 | 非公平 | 一般业务 |
| **ReentrantLock** | 中 | 中 | 可公平 | 需要 tryLock / Condition |
| **AtomicInteger** | 中 | 高 | 非公平 | 简单计数器 |

**老师点评**：

> "用 synchronized 和 ReentrantLock 是 **悲观锁**——我假设别人会抢，先锁住。  
> 用 AtomicInteger 是 **乐观锁**——我假设没人抢，先做，做失败再重试。  
> 多数场景乐观锁更快，但**ABA 问题**要注意。"

### ABA 问题（老师补充的）

```java
// 线程 A 读到 current = 100
// 线程 B 改成 99，再改回 100
// 线程 A CAS(100, 99) 成功！以为是原来那个 100
```

但实际上值被改过了又改回来。

**解决方案**：用 `AtomicStampedReference`，加个版本号：

```java
AtomicStampedReference<Integer> ref = 
    new AtomicStampedReference<>(100, 0);
// 每次改都要带版本号
ref.compareAndSet(100, 99, ref.getStamp(), ref.getStamp() + 1);
```

这次作业没用到，但**面试爱问**。

## 我做完这个实验的几点心得

### 1. 线程安全的本质

`synchronized` / `ReentrantLock` / `AtomicInteger` 都是为了**让"读-改-写"三步变原子**。

没有原子性，多线程就有竞态条件（race condition）。

### 2. 锁的粒度

- **锁太小**：包不住并发问题（V1 那种）
- **锁太大**：变成单线程（V3 那种）
- **合适粒度**：锁住"读-改-写"那段最小代码

### 3. sleep 是双刃剑

实验里加 `Thread.sleep(50)` 是为了模拟业务耗时，**也让其他线程有机会抢锁**。

但生产代码里，**sleep 不能用来协调线程**——应该用 `wait/notify`、`CountDownLatch` 等工具。

### 4. 性能调优

我跑了下三种方案的吞吐量（每秒钟卖多少张票）：

```
synchronized:      ~18 张/秒
ReentrantLock:     ~19 张/秒
AtomicInteger:     ~24 张/秒
```

乐观锁（CAS）比悲观锁快，**因为没有线程阻塞**。

但 CAS 自旋会**消耗 CPU**，高竞争场景反而慢。**实际项目里要压测**。

## 老师留的延伸题

> "如果用 `ConcurrentLinkedQueue` 装票，每个线程 `poll()` 一张，是不是天然线程安全？"

是的——因为 `ConcurrentLinkedQueue` 的 `poll()` 本身就是 CAS 原子操作。

```java
ConcurrentLinkedQueue<Integer> queue = new ConcurrentLinkedQueue<>();
for (int i = 1; i <= 100; i++) queue.offer(i);

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        while (true) {
            Integer ticket = queue.poll();
            if (ticket == null) break;
            System.out.println(Thread.currentThread().getName() 
                               + " 卖出第 " + ticket + " 号票");
            try { Thread.sleep(50); } catch (...) {}
        }
    }).start();
}
```

**完全不用自己加锁**——这才是 Java 并发包设计的优雅之处。
