## 起因

下学期开《计算机网络》，课前先自己撸一遍协议栈，免得上课一脸懵。

这篇记 OSI 七层模型、TCP/IP 四层模型，以及让我对"网络"祛魅的三次握手/四次挥手。

## 先分清两个模型

**OSI 七层**：学术派的标准模型，国际标准化组织写的。

**TCP/IP 四层**：实际互联网用的协议栈，工程派。

| OSI 七层 | TCP/IP 四层 | 对应协议 |
|----------|-------------|----------|
| 应用层 | **应用层** | HTTP, FTP, DNS, SMTP |
| 表示层 | | (合并到应用层) |
| 会话层 | | (合并到应用层) |
| 传输层 | **传输层** | TCP, UDP |
| 网络层 | **网络层** | IP, ICMP, ARP |
| 数据链路层 | **网络接口层** | 以太网, WiFi, PPP |
| 物理层 | | (合并到网络接口层) |

**为什么 TCP/IP 只有四层**：OSI 后两层（数据链路、物理层）在实际网络里紧耦合，没必要分。  
OSI 中间两层（表示、会话）在实际协议里要么不存在，要么和应用层揉在一起。

> 面试题：OSI 七层和 TCP/IP 四层的区别？——答 OSI 是理论模型，TCP/IP 是实际协议栈。

## 物理层：01 信号

光纤、电缆、无线电波——只管把 0/1 传过去。  
**不关心**：内容是啥、给谁。  
**关心**：电压、频率、编码方式。

跟我写代码关系不大，跳过。

## 数据链路层：MAC 地址 + 帧

**核心**：在**同一局域网**内，基于 MAC 地址转发"帧"。

MAC 地址是网卡出厂时烧死的 48 位地址，全球唯一：

```
00:1A:2B:3C:4D:5E
```

**帧的结构**（简化）：

```
| 目标 MAC | 源 MAC | 类型 | 数据 | CRC |
| 6 字节  | 6 字节 | 2   | 0-1500 | 4  |
```

**课堂容易搞混的点**：MAC 地址只在局域网内有效，跨路由器就找不到了。  
路由器会重新封装帧，把源/目标 MAC 改成自己的。

## 网络层：IP 地址 + 包

**核心**：在不同网络之间，基于 IP 地址转发"包"。

IPv4 地址：32 位，分成网络号和主机号：

```
192.168.1.100
└──┬──┘ └┬┘
 网络号  主机号
```

子网掩码决定哪些位是网络号：

```
255.255.255.0 → 前 24 位是网络号
```

**IP 包结构**（简化）：

```
| 版本 | TTL | 协议 | 源 IP | 目标 IP | 数据 |
```

**TTL（Time To Live）**：每经过一个路由器减 1，到 0 就丢弃。防止包永远在网络里转圈。  
**协议字段**：6 = TCP，17 = UDP。

## 传输层：TCP vs UDP

**TCP（Transmission Control Protocol）**：

- 面向连接（先握手建立连接）
- 可靠（确认 + 重传 + 序号）
- 有序（按序号重组）
- 流量控制（滑动窗口）

**UDP（User Datagram Protocol）**：

- 无连接（直接发）
- 不可靠（丢了不重传）
- 不保证顺序
- 但**快**（头部只有 8 字节）

**对比**：

| | TCP | UDP |
|--|-----|-----|
| 场景 | HTTP, 文件传输, 邮件 | 视频直播, DNS, 游戏 |
| 头部 | 20 字节 | 8 字节 |
| 连接 | 三次握手 | 无 |
| 可靠性 | 高 | 低 |
| 速度 | 慢 | 快 |

## TCP 三次握手（重点中的重点）

每次面试必问。

**目的**：客户端和服务端**互相确认**收发能力正常。

```
客户端                          服务端
  |                               |
  | --- SYN (seq=x) ---------->  |  第一次：客户端请求建立连接
  |                               |
  | <--- SYN+ACK (seq=y, ack=x+1) |  第二次：服务端同意，并确认收到
  |                               |
  | --- ACK (seq=x+1, ack=y+1) --> |  第三次：客户端确认
  |                               |
  |        连接建立！              |
```

**问题：为什么是三次不是两次？**

假设两次：

```
客户端              服务端
  | -- SYN -->        |
  |                   |
  | <-- SYN+ACK --    |  服务端认为连接建立
  |                   |  等客户端发数据
  | (但 SYN 丢了)     |  但客户端不知道服务端回了
  | 重新发 SYN        |  服务端收到又开一个连接
  ...                 → 浪费资源 + 数据错乱
```

**三次的关键**：

- 第一次：客户端证明"我能发"
- 第二次：服务端证明"我能收 + 我能发"
- 第三次：客户端证明"我能收"

只有来回都"打通过"，双方才都放心。

## TCP 四次挥手

关闭连接要四次，因为 TCP 是**全双工**（两个方向都能发），要分别关。

```
客户端                          服务端
  |                               |
  | --- FIN (seq=u) ---------->   |  第一次：客户端说"我没数据了"
  |                               |
  | <--- ACK (ack=u+1) ----------  |  第二次：服务端确认
  |                               |
  | <--- FIN (seq=v) ----------  |  第三次：服务端也说"我也没了"
  |                               |
  | --- ACK (ack=v+1) ---------->  |  第四次：客户端确认
  |                               |
  |        连接关闭！              |
```

**问题：为什么最后客户端要等 2MSL（Maximum Segment Lifetime）？**

**答**：确保最后一个 ACK 能到达服务端。  
如果 ACK 丢了，服务端会重发 FIN，客户端必须能收到再回 ACK。  
2MSL = 一个报文在网络里最长的存活时间，过了这么久都没重发，说明 ACK 真的到了。

## 抓个包看看（Windows 上）

用 Wireshark 抓一个 HTTP 请求：

```bash
# 1. 下载 Wireshark: https://www.wireshark.org/
# 2. 选网卡（一般选 WiFi）
# 3. 过滤器输入：tcp.port == 80
# 4. 浏览器访问 http://example.com
```

看到的结果大致是：

```
1. [SYN] Seq=0                    ← 三次握手第 1 次
2. [SYN, ACK] Seq=0 Ack=1         ← 第 2 次
3. [ACK] Seq=1 Ack=1              ← 第 3 次
4. [PSH, ACK] GET / HTTP/1.1      ← 应用层 HTTP 请求
5. [ACK]                          ← 服务端确认收到请求
6. [PSH, ACK] HTTP/1.1 200 OK ... ← 服务端响应
7. [FIN, ACK]                    ← 四次挥手开始
8. ...
```

第一次看到这个流程图的时候，我才真正理解"网络协议"不是黑魔法——就是**两端按约定的格式发字节流**。

## 几个常考小点

### 1. 在浏览器输入 URL 发生了什么？

按 OSI 分层：

1. **应用层**：DNS 查询域名 → IP；HTTP 构造请求
2. **传输层**：TCP 三次握手建立连接
3. **网络层**：IP 包封装，加源/目标 IP
4. **数据链路层**：ARP 解析 IP → MAC，加 MAC 头
5. **物理层**：网卡发送比特流

然后服务端反向走一遍回来。

### 2. HTTPS vs HTTP

HTTPS = HTTP + TLS（加密）。  
TLS 握手在 TCP 握手之后，又来一次：

```
TCP 三次握手 → TLS 4 次握手 → HTTP 请求
```

### 3. UDP 怎么实现可靠传输？

经典面试题。**答**：在应用层加确认、重传、序号——比如 QUIC 协议就是 UDP + 应用层可靠性实现的。
