## 起因

软件测试课第一次作业：

> "为一个用户登录功能编写测试用例，要求用等价类划分法，至少 10 条。"

我之前对"测试"的理解就是"手动点一点，看会不会报错"。看了课本才知道，**测试是有方法论的**——等价类划分就是其中最基础的一种。

这篇把整个"从需求到用例"的过程记下来。

## 需求：登录功能

老师给的登录功能描述：

```
输入：用户名、密码
处理：
  1. 用户名为空 → 提示"用户名不能为空"
  2. 用户名长度 < 3 或 > 20 → 提示"用户名长度 3-20"
  3. 用户名包含特殊字符 → 提示"用户名只能包含字母数字下划线"
  4. 密码为空 → 提示"密码不能为空"
  5. 密码长度 < 6 或 > 20 → 提示"密码长度 6-20"
  6. 用户名不存在 → 提示"用户名或密码错误"
  7. 密码错误 → 提示"用户名或密码错误"
  8. 校验通过 → 跳转到首页
```

## 第一步：识别输入条件

把需求里的"输入条件"一条条拆出来：

| 编号 | 条件 | 有效等价类 | 无效等价类 |
|------|------|-----------|-----------|
| 1 | 用户名 | 3-20 字符 | 空、`&lt;3`、`&gt;20`、特殊字符 |
| 2 | 密码 | 6-20 字符 | 空、`&lt;6`、`&gt;20` |

## 第二步：画等价类表

### 用户名

| 编号 | 输入条件 | 有效/无效 | 典型数据 |
|------|----------|-----------|----------|
| U1 | 用户名 = 空 | 无效 | `""` |
| U2 | 用户名长度 = 3 | 有效 | `"abc"` |
| U3 | 用户名长度 = 20 | 有效 | `"a2345678901234567890"`（20位）|
| U4 | 用户名长度 = 2 | 无效 | `"ab"` |
| U5 | 用户名长度 = 21 | 无效 | 21 位字符串 |
| U6 | 用户名含特殊字符 | 无效 | `"abc@"` |
| U7 | 用户名只有字母数字下划线 | 有效 | `"abc_123"` |

### 密码

| 编号 | 输入条件 | 有效/无效 | 典型数据 |
|------|----------|-----------|----------|
| P1 | 密码 = 空 | 无效 | `""` |
| P2 | 密码长度 = 6 | 有效 | `"123456"` |
| P3 | 密码长度 = 20 | 有效 | 20 位 |
| P4 | 密码长度 = 5 | 无效 | `"12345"` |
| P5 | 密码长度 = 21 | 无效 | 21 位 |

## 第三步：写测试用例

### 用例编号规则

`TC-<功能>-<编号>`，比如登录的第一个用例叫 `TC-LOGIN-001`。

### 有效等价类用例

#### TC-LOGIN-001：合法的用户名 + 合法的密码

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `abc_123`，密码 `123456` |
| **预期** | 登录成功，跳转到首页 |
| **覆盖等价类** | U2/U3/U7 + P2/P3 |

#### TC-LOGIN-002：边界值 — 用户名最短合法

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `abc`（3 字符），密码 `123456` |
| **预期** | 登录成功 |
| **覆盖等价类** | U2（边界）+ P2 |

#### TC-LOGIN-003：边界值 — 用户名最长合法

| 项 | 内容 |
|----|------|
| **输入** | 用户名 20 字符，密码 `123456` |
| **预期** | 登录成功 |
| **覆盖等价类** | U3（边界）+ P2 |

### 无效等价类用例（每条单独测一个无效）

**为什么无效等价类要分开测？**—— 因为系统对每条无效的处理可能不同，**必须确认每条都触发了正确的错误提示**。

#### TC-LOGIN-004：用户名为空

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `""`，密码 `123456` |
| **预期** | 提示"用户名不能为空"，不发起后端请求 |
| **覆盖等价类** | U1 |

#### TC-LOGIN-005：用户名太短

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `"ab"`，密码 `123456` |
| **预期** | 提示"用户名长度 3-20" |
| **覆盖等价类** | U4 |

#### TC-LOGIN-006：用户名太长

| 项 | 内容 |
|----|------|
| **输入** | 用户名 21 字符，密码 `123456` |
| **预期** | 提示"用户名长度 3-20" |
| **覆盖等价类** | U5 |

#### TC-LOGIN-007：用户名含特殊字符

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `"abc@"`，密码 `123456` |
| **预期** | 提示"用户名只能包含字母数字下划线" |
| **覆盖等价类** | U6 |

#### TC-LOGIN-008：密码为空

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `abc_123`，密码 `""` |
| **预期** | 提示"密码不能为空" |
| **覆盖等价类** | P1 |

#### TC-LOGIN-009：密码太短

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `abc_123`，密码 `"12345"` |
| **预期** | 提示"密码长度 6-20" |
| **覆盖等价类** | P4 |

#### TC-LOGIN-010：密码太长

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `abc_123`，密码 21 字符 |
| **预期** | 提示"密码长度 6-20" |
| **覆盖等价类** | P5 |

#### TC-LOGIN-011：用户名不存在

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `"nonexistent"`，密码 `123456` |
| **预期** | 提示"用户名或密码错误" |
| **说明** | 注意：**不提示"用户名不存在"**，避免泄露系统信息 |

#### TC-LOGIN-012：密码错误

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `"alice"`（已注册），密码 `"wrongpwd"` |
| **预期** | 提示"用户名或密码错误" |
| **说明** | 同上，不区分"用户存在但密码错"和"用户不存在"，防暴力枚举 |

#### TC-LOGIN-013：用户名 + 密码都无效（验证优先级）

| 项 | 内容 |
|----|------|
| **输入** | 用户名 `""`，密码 `""` |
| **预期** | 先提示"用户名不能为空"（按需求顺序判断）|
| **说明** | 验证错误提示的优先级 |

## 我写完之后的反思

### 1. 等价类划分的关键是"代表性"

**每个等价类里随便挑一个数据就行**，因为同一个等价类的数据行为应该一致。

比如"用户名长度 < 3"这个无效等价类，挑 `"a"` `"ab"` 都一样。

### 2. 边界值要单独测

`长度 3` 和 `长度 2` 是两个等价类，但 **`长度 3` 是边界**——最容易出 bug。

我特地为最短合法值（3）和最长合法值（20）写了用例。

### 3. 错误信息不能泄露系统信息

`用户名不存在` vs `密码错误`——**统一提示"用户名或密码错误"**，否则攻击者可以枚举用户名。

这一条老师没要求，**我自己加的**。交作业时老师给了好评："考虑到了安全性"。

### 4. 测试用例的"颗粒度"

老师点评时提到一个我之前没想到的点：

> "无效等价类每个一条用例，但**有效等价类可以合并**——多个有效条件同时满足就行。"

所以 TC-LOGIN-001 把用户名和密码的有效等价类**合并成一个用例**，而不是拆 4 个。

## 等价类划分的步骤总结

1. **识别输入条件**——从需求里找所有"输入"和"约束"
2. **划分有效/无效等价类**——符合需求 vs 不符合
3. **给每个等价类选代表数据**——边界值优先
4. **写用例**——有效等价类**尽量合并**，无效等价类**每个单独一条**
5. **执行用例**——记录实际结果和预期对比
6. **跟踪 bug**——发现的不符合写到 bug 列表

## 一个我犯过的低级错误

第一次交作业时，我把"用户名 = 3 字符"和"用户名 = 20 字符"**当成两个等价类**：

```
U2: 用户名长度 = 3   ← 有效
U3: 用户名长度 = 20  ← 有效
```

但其实它们是**同一个等价类**（"用户名长度在 3-20 之间"），只是边界值。

**正解**：

```
U2: 用户名长度在 3-20 之间   ← 有效（用一个用例覆盖）
U2-boundary-low: 用户名 = 3  ← 边界用例
U2-boundary-high: 用户名 = 20 ← 边界用例
```

`U3` 是冗余的，等价类表合并掉。
