## 起因

学索引的时候背了一堆规则：

- "最左前缀"
- "范围查询右边失效"
- "LIKE 以 % 开头不走索引"

但**背下来和真的搞懂是两码事**。

正好准备面试，想拿真实数据跑一遍 Explain，看每条 SQL 实际怎么执行的。这篇是完整的实验过程和结果。

## 准备工作：建一张 50 万行的表

我有一张老的 `t_order` 表，**只有 1000 条**，跑 Explain 看不出区别（都全表扫都很快）。

手动造 50 万条数据：

```sql
CREATE TABLE t_order (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT,
    product_id BIGINT,
    status TINYINT,
    amount DECIMAL(10, 2),
    created_at DATETIME,
    remark VARCHAR(100)
);

-- 插 50 万条（用 INSERT ... SELECT 加速）
INSERT INTO t_order (user_id, product_id, status, amount, created_at, remark)
SELECT
    FLOOR(RAND() * 10000),           -- user_id: 0~9999
    FLOOR(RAND() * 1000),            -- product_id: 0~999
    FLOOR(RAND() * 4),               -- status: 0~3
    ROUND(RAND() * 1000, 2),         -- amount
    DATE_ADD('2024-01-01', INTERVAL FLOOR(RAND() * 365) DAY),
    CONCAT('order_', UUID())
FROM information_schema.columns a, information_schema.columns b
LIMIT 500000;  -- 大概需要 30 秒
```

跑完查询下：

```sql
SELECT COUNT(*) FROM t_order;
-- 500000
```

数据准备好了。

## 实验 1：没建索引时

先建一个**没索引**的版本（主键除外）：

```sql
-- 只有主键索引
SHOW INDEX FROM t_order;
```

```
+----------+------------+----------+--------------+-------------+
| Key_name | Column_name| Non_unique| Index_type   |
+----------+------------+----------+--------------+-------------+
| PRIMARY  | id         | 0        | BTREE        |
+----------+------------+----------+--------------+-------------+
```

跑一条查询：

```sql
EXPLAIN SELECT * FROM t_order WHERE user_id = 12345;
```

**输出**：

```
+----+-------------+---------+------+---------------+------+---------+------+--------+-------------+
| id | select_type | table   | type | possible_keys | key  | rows    | Extra |
+----+-------------+---------+------+---------------+------+---------+------+--------+-------------+
|  1 | SIMPLE      | t_order | ALL  | NULL          | NULL | 498532  | Using where |
+----+-------------+---------+------+---------------+------+---------+------+--------+-------------+
```

**关键信息**：

- `type: ALL` —— **全表扫描**
- `rows: 498532` —— 扫描了 50 万行
- `key: NULL` —— 没用到索引
- `Extra: Using where` —— 在 server 层做 where 过滤

实际执行：

```sql
SELECT BENCHMARK(1, (SELECT * FROM t_order WHERE user_id = 12345));
-- 时间：~0.45 秒
```

**0.45 秒**，对一个查询来说**慢得不能接受**。

## 实验 2：单列索引

加个单列索引：

```sql
CREATE INDEX idx_user_id ON t_order(user_id);
```

再跑同样的查询：

```sql
EXPLAIN SELECT * FROM t_order WHERE user_id = 12345;
```

**输出**：

```
+----+-------------+---------+------+---------------+-------------+---------+-------+-------------+
| id | table       | type    | key           | rows   | Extra              |
+----+-------------+---------+------+---------------+-------------+---------+-------+-------------+
|  1 | t_order     | ref     | idx_user_id   | 49     | NULL               |
+----+-------------+---------+------+---------------+-------------+---------+-------+-------------+
```

**关键信息**：

- `type: ref` —— **非唯一索引等值查询**
- `rows: 49` —— 只扫 49 行
- `key: idx_user_id` —— 用上了索引
- `Extra: NULL` —— 没额外的 where 过滤

实际执行：

```
时间：~0.003 秒（3 毫秒）
```

**从 0.45 秒降到 0.003 秒**——提速 **150 倍**。

## 实验 3：联合索引（最左前缀）

删掉单列索引，建联合索引：

```sql
DROP INDEX idx_user_id ON t_order;
CREATE INDEX idx_user_status ON t_order(user_id, status);
```

跑查询 A（**符合**最左前缀）：

```sql
EXPLAIN SELECT * FROM t_order 
WHERE user_id = 12345 AND status = 1;
```

**输出**：

```
type: ref
key: idx_user_status
rows: 12
Extra: NULL
```

✅ 完美命中。

跑查询 B（**跳过 user_id**）：

```sql
EXPLAIN SELECT * FROM t_order 
WHERE status = 1;
```

**输出**：

```
type: ALL
key: NULL
rows: 498532
Extra: Using where
```

❌ **全表扫描**——因为跳过了 `user_id`，联合索引失效。

这就是"最左前缀"的字面意思：**必须从最左边的列开始用，不能跳过**。

## 实验 4：范围查询右边失效

```sql
-- 还是这个联合索引
-- idx_user_status: (user_id, status)

EXPLAIN SELECT * FROM t_order 
WHERE user_id = 12345 AND status > 0;
```

**输出**：

```
type: range
key: idx_user_status
rows: 49
Extra: Using index condition
```

`status > 0` 是范围查询，**`status` 自己用了索引**（变 `range`），但如果还有第三列 `created_at`，那个就废了。

加一列试试：

```sql
CREATE INDEX idx_user_status_created ON t_order(user_id, status, created_at);
DROP INDEX idx_user_status ON t_order;

EXPLAIN SELECT * FROM t_order 
WHERE user_id = 12345 
  AND status > 0 
  AND created_at > '2024-06-01';
```

**输出**：

```
type: range
key: idx_user_status_created
rows: 8
Extra: Using index condition
```

- `user_id` 等值 → 命中
- `status > 0` 范围 → 用上索引（变 range）
- `created_at > '2024-06-01'` → **失效**（因为在 status 后面）

`Using index condition` 是 ICP（Index Condition Pushdown），MySQL 5.6+ 会把 where 条件下推到索引层，**减少回表**。

## 实验 5：LIKE 的 % 位置

```sql
-- 还是 idx_user_status_created: (user_id, status, created_at)

-- LIKE '%abc%' (前导 %)
EXPLAIN SELECT * FROM t_order WHERE remark LIKE '%order%';
```

**输出**：

```
type: ALL
key: NULL
rows: 498532
Extra: Using where
```

❌ **全表扫**——因为 `remark` 根本不在索引里。

换查询条件，让它命中索引：

```sql
-- 假设我们建了 remark 的索引
CREATE INDEX idx_remark ON t_order(remark);

EXPLAIN SELECT * FROM t_order WHERE remark LIKE 'order%';
```

**输出**：

```
type: range
key: idx_remark
rows: 12453
Extra: Using where
```

✅ **前导匹配**能用上索引。

`EXPLAIN SELECT * FROM t_order WHERE remark LIKE '%order%';`

```
type: ALL
key: NULL
rows: 498532
Extra: Using where
```

❌ **% 在前面**不走索引。

这是 B+ 树的结构决定的：**字符串比较必须从头开始**。

## 实验 6：函数让索引失效

```sql
-- 我们有 idx_created_at? 没有，建一个
CREATE INDEX idx_created_at ON t_order(created_at);

-- 查询 1：直接用索引列
EXPLAIN SELECT * FROM t_order 
WHERE created_at > '2024-06-01';
```

**输出**：

```
type: range
key: idx_created_at
rows: 213456
```

✅ 命中。

```sql
-- 查询 2：对索引列用函数
EXPLAIN SELECT * FROM t_order 
WHERE YEAR(created_at) = 2024;
```

**输出**：

```
type: ALL
key: NULL
rows: 498532
Extra: Using where
```

❌ **全表扫**——因为 `YEAR(created_at)` 是个函数，MySQL 不知道函数结果和索引值的关系，**索性全扫**。

**正解**：

```sql
-- 改成范围查询，让索引生效
EXPLAIN SELECT * FROM t_order 
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

## 实验 7：覆盖索引（不用回表）

联合索引 `(user_id, status, created_at)` 里查询时**只查索引列**：

```sql
EXPLAIN SELECT user_id, status, created_at FROM t_order 
WHERE user_id = 12345;
```

**输出**：

```
type: ref
key: idx_user_status_created
rows: 49
Extra: Using index    ← 关键！
```

`Using index` 表示**不需要回表**——查询的列都在索引里，MySQL 直接从索引拿数据，**不用回到主键索引**。

这叫**覆盖索引**，**比回表快很多**。

## 实验 8：ORDER BY 是否走索引

```sql
-- 没索引时排序
EXPLAIN SELECT * FROM t_order ORDER BY created_at DESC LIMIT 10;
```

**输出**：

```
type: ALL
key: NULL
rows: 498532
Extra: Using filesort   ← 坏消息！
```

`Using filesort` 意味着 MySQL 要在内存里排序，**慢**。

加索引：

```sql
CREATE INDEX idx_created_at_desc ON t_order(created_at);
```

注意 MySQL 8.0+ 支持降序索引，普通索引对 ORDER BY DESC 也有效：

```sql
EXPLAIN SELECT * FROM t_order ORDER BY created_at DESC LIMIT 10;
```

**输出**：

```
type: index
key: idx_created_at_desc
rows: 10
Extra: Using where     ← 没 filesort 了！
```

`type: index` 表示**全索引扫描**（比全表扫快，因为索引比数据小），但只扫前 10 条。

## 跑完所有实验的总结表

我把 7 个实验的 Explain 结果汇总：

| 实验 | SQL 关键片段 | type | key | rows | Extra |
|------|--------------|------|-----|------|-------|
| 1 无索引 | `WHERE user_id=12345` | ALL | NULL | 50w | Using where |
| 2 单列索引 | 同上 | ref | idx_user_id | 49 | NULL |
| 3a 最左前缀 | `user_id=12345 AND status=1` | ref | idx_user_status | 12 | NULL |
| 3b 跳过 user_id | `status=1` | ALL | NULL | 50w | Using where |
| 4 范围后失效 | `user_id=12345 AND status>0 AND created_at>...` | range | idx_user_status_created | 8 | ICP |
| 5 LIKE 前导 | `remark LIKE 'order%'` | range | idx_remark | 12k | Using where |
| 5 LIKE 中间 | `remark LIKE '%order%'` | ALL | NULL | 50w | Using where |
| 6 函数失效 | `YEAR(created_at)=2024` | ALL | NULL | 50w | Using where |
| 7 覆盖索引 | `SELECT user_id, status` | ref | idx_... | 49 | Using index |
| 8 排序 | `ORDER BY created_at DESC` | index | idx_created_at | 10 | Using where |

**规律总结**：

1. **`type` 从好到坏**：`system > const > eq_ref > ref > range > index > ALL`
2. **理想状态**：`type=ref/range`，`key` 不为 NULL，`rows` 很小
3. **`Using filesort` 和 `Using temporary` 是坏信号**——尽量优化掉
4. **`Using index` 是好信号**——覆盖索引，不用回表

## 一个我没搞懂的坑

我用 `IN` 的时候遇到一个问题：

```sql
EXPLAIN SELECT * FROM t_order WHERE user_id IN (1, 2, 3, 4, 5);
```

输出：

```
type: range
key: idx_user_id
rows: 250
```

✅ 正常。

但如果 `IN` 里的值很多（> 某个阈值）：

```sql
EXPLAIN SELECT * FROM t_order WHERE user_id IN (1,2,3,...,2000);
```

输出会变成：

```
type: ALL
key: NULL
rows: 500000
Extra: Using where
```

❌ **IN 太多时 MySQL 放弃索引**——它觉得全表扫更快。

具体阈值是多少，**取决于数据分布**（IN 命中的行占总行数的比例）。我这次大概 2000 左右开始放弃。

**实际项目经验**：`IN` 里的值**控制在 100 以内**比较稳，超过用临时表或者拆成多次查询。

## 给学索引的朋友的建议

1. **别背规则，自己跑一遍**——Explain 输出看一眼胜过读 10 篇文章
2. **造数据用 `INSERT ... SELECT FROM information_schema`**——比写脚本快
3. **`type` 是最重要的字段**——看一眼就知道查询走没走索引
4. **覆盖索引是真神**——查询列都在索引里，性能飙升
5. **生产前一定要压测**——实验室的结果不一定等于生产
