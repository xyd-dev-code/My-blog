## 起因

半年前 Karpathy 在 X 上发了那条推文,造了"vibe coding"这个词:

> "我接受代码不是手写。 我只是看着东西,说'加个渐变'、'把 padding 改大',然后跑起来。 代码我自己都不读。"

当时评论区两极分化:一派说"未来已来",一派说"这就是技术债的来源"。

半年过去,各家 IDE 都接了 AI 助手(Claude Code / Cursor / Cline / Copilot),原型生成工具也起来了(v0 / bolt.new / Lovable / Replit Agent)。

我用了一段时间,也读了一些社区反馈和 Simon Willison 的几篇博客,有些判断想沉淀下来。

**这篇不是产品评测**,也不是"AI 取代程序员"的讨论,而是**几个观察**——半年下来,真正改变工作流的不是营销话术里那些东西,反而是几个被低估的细节。

## 观察一:AI 出的代码,调试起来比手写的累

这是社区里反复出现的一个声音,我自己也中过几次。

手写代码 bug,debug 思路是:

1. 知道这段代码做了什么
2. 推演可能出错的地方
3. 加 `console.log` 验证
4. 改

AI 生成的代码 bug,debug 思路是:

1. **先得搞清楚这段代码到底做了什么**(因为不是自己写的)
2. 怀疑 AI 的"思路"本身就有问题(可能用了错模式)
3. 不敢轻易改 — 改完可能引入新的 AI 风味代码
4. 越改越长,直到把"我本来要改什么"都忘了

**根因**:**你跟代码的"心智距离"不一样了**。

手写代码,你在写的那 30 秒里已经把约束、命名、调用关系在脑子里过了一遍。AI 写代码,你看的是"成品",约束在 prompt 里、命名在 AI 的训练分布里、调用关系在 AI 的"我会怎么写"里。

**有一类 bug 特别难调**:AI 用了**对的语法、错的语义**。比如它写了一个 `isValidUser`,在某些边界情况下返回 `true`,在另一些边界情况下也返回 `true`,**但语义上是错的**。这种 bug 阅读代码看不出来,跑测试能跑过(因为没写那个 case),只有真实用户用了才崩。

社区有个词叫"**看起来对的代码**"(plausible code) — AI 特别擅长生成这种,调试起来也最难。

## 观察二:上下文会腐烂

我读 Simon Willison 的博客,他提了一个词:**context rot**。

意思是:**会话越长,AI 对前面对话的"记忆"越模糊**。

不是技术意义上的"上下文窗口"满了,而是 AI 实际**遵循**前面约束的能力下降。

我自己的体感:

- **前 10 轮对话**:AI 严格按你之前说的"命名风格 / 不用某个库 / 错误处理用 X" 写
- **30 轮以后**:AI 开始"我行我素" — 用了一个你明确说过不要的库,命名风格飘了,突然忘了前面的约定
- **100 轮以后**:AI 完全 fresh start,像换了一个 agent

**为什么会这样**:

1. 上下文窗口里的"约束"被后续对话稀释了 — 100 条消息里,约束只占 5%,AI 注意力分散
2. AI 的"工作记忆"不是无限 — 跟人一样,新东西进来,旧东西被压缩
3. 模型本身没有"project 概念" — 它看到的就是 token 流,不知道哪些是约束、哪些是任务、哪些是闲聊

**应对方法**(社区经验):

| 策略 | 怎么做 | 适用 |
|------|------|------|
| **短会话** | 任务分块,每块开新会话 | 持续 2 小时+ 的工作 |
| **写 memory 文件** | 把约束写成 markdown,每段会话开始时 AI 先读 | 长期项目 |
| **用子 agent** | 复杂任务拆给 subagent,主 agent 保持短上下文 | 多文件改动 |
| **结构化 prompt** | "约束" 单独一段,不和任务混 | 一次性任务 |

我之前维护一个长期项目时用的就是 **"memory 文件 + 短会话"**:项目根目录放一个 `MEMORY.md`,AI 接手任务前先读一遍;单个任务超过 1 小时就开新会话,把上次的成果写进 memory。

## 观察三:v0 / bolt.new / Lovable 跟 IDE 集成不是同一种东西

这是被混淆最多的一个点。

**两类工具的输入输出完全不一样**:

```
┌─────────────────────────┬──────────────────────────────┐
│  IDE 集成 (Claude Code) │  原型生成 (v0 / bolt.new)    │
├─────────────────────────┼──────────────────────────────┤
│  输入: 你的代码库       │  输入: 一段自然语言描述      │
│  输出: 修改/新增文件    │  输出: 完整可运行原型        │
│  上下文: 你的项目结构   │  上下文: 它的云端环境        │
│  适合: 增量改动         │  适合: 0 → 1 原型            │
│  局限: 一次改 1-3 个文件│  局限: 难接入真实项目        │
└─────────────────────────┴──────────────────────────────┘
```

**v0 / bolt.new / Lovable** 干的事是"**用 prompt 出一个能跑的网站**" — 你说"做一个记账的 web app,带登录",它给你一个能部署的 Next.js + Tailwind 项目,带基本功能。

**Claude Code / Cursor** 干的事是"**在已有项目里改文件**" — 你说"给这个函数加个缓存",它在你项目里 `Edit` 一个文件,改完你 `pnpm test` 跑一遍。

**为什么这两类不能简单替换**:

- 拿 v0 给生产项目加一个页面?可以,但生成完是**新项目**的代码,跟你的**旧项目**架构、命名、测试、CI 都不一样。要接入?得手动合并。这是反 vibe coding 的 — vibe coding 承诺"我不读代码",合并就需要读了。
- 拿 Claude Code 从 0 起一个项目?能,但你会**输一万字 prompt**描述你要的架构、库、命名约定,然后 AI 慢慢输出 50 个文件。v0 一句话搞定。这就是不同工具的擅长区。

**实际的混搭流**(社区常见):

1. **v0 出原型** — 5 分钟,有个能跑的东西
2. **看效果,聊需求** — 人脑决定改什么
3. **Claude Code 接入真项目** — 把 v0 的代码 merge 进真项目
4. **继续维护** — Claude Code 改 bug、加功能

**反例**(也常见):

1. 用 v0 做生产项目 → 维护地狱,v0 不能改生产代码
2. 用 Claude Code 从 0 起项目 → 太慢,不如手写
3. 把 v0 的代码直接复制到生产项目 → 风格不一致,要重写

**判断用哪个的简单标准**:

| 场景 | 用什么 |
|------|------|
| **画 UI 原型 / 验证想法** | v0 / bolt.new |
| **客户演示 / hackathon 演示** | v0 / Lovable |
| **接入现有项目改 bug** | Claude Code / Cursor |
| **生产代码长期维护** | Claude Code / Cursor |
| **学习新库、试 API** | v0(快速跑通)+ Cursor(深入集成) |
| **写脚本 / 一次性工具** | v0(快) |

## 观察四:AI 真做得比人好的,是三件"非代码"的事

把"AI 写代码"这个笼统说法拆开,**AI 在不同任务上的表现差很多**:

**强项**(AI 明显比人快):

1. **写样板代码** — CRUD、表单验证、API 端点、配置文件,模式重复,AI 写得快且准
2. **跨文件改动** — "把这个命名在所有文件里改掉" / "把所有 API 调用加上重试",人做要 10 分钟,AI 做要 10 秒
3. **读陌生代码** — 接手别人写的、或者很久前自己写的代码,AI 能快速总结"这文件干啥" / "这个类负责什么"
4. **解释 API / 库用法** — 比查文档快,因为能直接给"在你这个上下文里怎么用"
5. **写测试** — 给定一个函数,写一组覆盖正常 / 边界 / 异常的测试,AI 写得比人快 5-10 倍

**弱项**(AI 经常翻车):

1. **架构决策** — 用什么模式 / 怎么分模块 / 怎么权衡,AI 给的建议需要人筛
2. **业务规则** — 哪些边界要处理 / 哪些状态要禁止,AI 不知道
3. **性能调优** — AI 能给"通常的做法",但具体到你的数据规模、硬件、QPS,得人做 profile
4. **复杂 bug 调试** — 涉及多系统交互、时序、并发,AI 看不出
5. **可读性 / 长期维护** — AI 写一次能跑,人看 6 个月后改不动

**关键的认知**:**强项里"非代码"占 3 个** — 样板代码、跨文件改动、读代码都不是"创意"工作,是**模式识别** + **机械操作**。AI 最擅长这类。

**弱项里"非代码"也占 3 个** — 业务规则、复杂 bug、长期维护都是**需要理解上下文、做判断**的事。AI 弱的是判断力,不是执行力。

**所以实际工作流**:

- AI 干**模式化**的事(样板 / 跨文件 / 读 / 写测试)
- 人干**判断**的事(架构 / 业务规则 / 性能 / 复杂 bug)
- **人保留"否决权"** — AI 出的代码人读一遍,不对劲就打回

## 观察五:半年下来,真正改变的事

半年用下来,我对"vibe coding"这个词的体感是这样的:

**没改变的事**:

- 写生产代码还是得人写 — AI 是协作者,不是代工
- 复杂 bug 还是得人调 — AI 帮不上
- 架构决策还是得人想 — AI 给的是"通常做法",不是"你这个项目的做法"
- 软件工程的核心能力(读代码、抽象、命名)还是核心 — AI 改变的是"做的速度",不是"做的方式"

**真正改变的事**:

1. **写样板代码不痛苦了** — CRUD / 表单 / API 端点,以前 1 小时,现在 5 分钟
2. **跨文件改动不烦了** — 重命名 / 加日志 / 改 API 签名,以前 20 分钟,现在 1 分钟
3. **接手新项目快 3 倍** — AI 总结代码、画架构图,以前看 1 天,现在看 1 小时
4. **学习新库的速度快很多** — 不用看完文档,直接问 AI"在我这个场景下怎么用"
5. **写测试不拖延了** — 之前总拖,现在 AI 一写就是一组
6. **不再写 README 写到半夜** — AI 写第一稿,我改

**心态变化**:

- 看到 fail 不焦虑 — 因为"3 步闭环"几乎一定能定位
- 不再追求"我一个人写完所有代码" — 接受"AI 写样板,我写逻辑"
- 写代码时间变少,想代码时间变多 — 反而是好事

## 反思:vibe coding 的"边界"在哪

社区讨论里有个问题反复出现:"什么时候 vibe coding 会变成 vibe debugging"?(vibe debug = AI 修 AI 写的 bug,越来越乱)

**我的判断**:**vibe coding 有个甜蜜区,越过就崩**。

```
甜蜜区(AI 主导):
- 一次性脚本
- 原型 / 演示
- 内部工具
- 学习项目

过渡区(人 + AI 协作):
- 长期项目新功能
- 重构
- bug 修复

非甜蜜区(人主导,AI 辅助):
- 核心算法
- 高性能代码
- 安全敏感代码
- 大型生产系统的关键模块
```

**判断标准简单**:**这个项目如果出 bug,会不会有人受伤?**

- **不会**(个人项目 / 内部工具 / 原型)→ vibe coding 随便用
- **会**(生产系统 / 用户数据)→ 人要严格审
